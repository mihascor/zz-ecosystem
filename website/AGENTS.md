# AGENTS.md

## Роль в проекте

Вы работаете в качестве помощника программиста в этом репозитории.
Не придумывайте требования к продукту. Сначала следуйте документации в /docs.

Перед внесением изменений:

- прочтите соответствующую документацию в /docs;
- проверьте существующие файлы;
- кратко объясните, что вы собираетесь изменить;
- внесите минимальное безопасное изменение.

## Основные проектные документы

Используйте эти файлы в качестве источника достоверной информации:

- docs/01-project-brief.md — цель продукта и пользователи
- docs/02-requirements.md — функциональные требования
- docs/03-architecture.md — архитектура системы
- docs/04-data-model.md — сущности и связи базы данных (если используется)
- docs/05-api-contracts.md — контракты API (если используются)
- docs/06-development-plan.md — этапы реализации
- docs/07-decisions.md — принятые технические решения

Если требование неясно:

- не гадайте молча;
- явно укажите предположение;
- предпочитайте минимальную реализацию;
- обновите docs/07-decisions.md, если принято техническое решение.

## Стиль разработки

Работайте поэтапно.

Не переписывайте большие части проекта, если это явно не требуется.

Предпочитайте небольшие коммиты/небольшие изменения.
Сохраняйте существующую архитектуру и соглашения об именовании.

После каждой задачи сообщайте:

- какие файлы были изменены;
- почему они были изменены;
- как проверить результат;
- что еще предстоит сделать.

## Правила безопасности

Не добавляйте в репозиторий секреты, токены, пароли, ключи API, cookie или закрытые учетные данные.
Не прописывайте учетные данные для производственной среды.
Используйте переменные окружения для конфигурации.
Если существует `.env.example`, обновляйте его при необходимости добавления новых переменных.

## Правила архитектуры

Уважайте выбранную архитектуру.

Не добавляйте новые фреймворки или тяжелые зависимости без одобрения.

Если новая зависимость кажется необходимой, объясните:

- почему она необходима;
- какие существуют альтернативы;
- какой риск она добавляет.

## Качество кода

Делайте код простым и читаемым.
Предпочитайте явные имена сложным абстракциям.
Добавляйте комментарии только там, где они разъясняют неочевидную логику.
Не усложняйте функции раннего MVP.

## Тестирование и валидация

После внесения изменений в код:

- запускайте доступные тесты, если они существуют;
- запускайте проверку линтера/типов, если она настроена;
- если тесты отсутствуют, объясните, что следует протестировать вручную.
- не запускайте визуальные проверки если не было отдельного требования

Не утверждайте, что что-то работает, если это не было проверено.
Если вы не можете запустить проверку, четко укажите это.

## Синхронизация документации

При изменении поведения реализации обновите соответствующий документ в /docs.
При принятии технического решения добавьте его в docs/07-decisions.md.

## Чистота репозитория

Не добавляйте в Git генерируемые, временные и локальные файлы:

- кэши и результаты сборки;
- логи;
- настройки IDE;
- файлы операционной системы;
- локальные данные и загрузки;
- служебные файлы тестовых инструментов;
- файлы окружения с секретами.

Перед завершением задачи:

1. Выполните `git status --short`.
2. Проверьте каждый новый файл.
3. Если файл генерируется автоматически и не является исходным кодом, миграцией, документацией, тестом или необходимой конфигурацией — добавьте его в `.gitignore`.
4. Не добавляйте файл в Git только потому, что он используется локально.
5. Не удаляйте уже отслеживаемые файлы без проверки их назначения.
6. Не считайте принцип «не участвует в production» единственным критерием: тесты, документация, миграции, скрипты разработки и примеры конфигурации должны храниться в репозитории.
