## Проект: сайт «Запретные знания»

## 1. Общая информация

### 1.1. Название проекта

**«Запретные знания»**

### 1.2. Тип проекта

Контентный сайт-блог для публикации статей, изображений и структурированного чтения материалов.

### 1.3. Назначение сайта

Сайт предназначен для размещения авторских материалов заказчика в формате статей. Пользователь должен иметь возможность читать материалы последовательно, выбирать статьи по категориям, подкатегориям и тегам, а также пользоваться удобной навигацией через левый сайдбар.

### 1.4. Формат разработки

Проект разрабатывается **локально**

---

## 2. Место проекта в общей экосистеме

Сайт является отдельным самостоятельным проектом внутри общей папки экосистемы.

Пример структуры:

```text
zz-ecosystem/
├─ website/              # сайт блог «Запретные знания»
├─ rag-system/           # будущая RAG-система
├─ ai-agent-chat/        # будущий ИИ-чат
├─ video-generator/      # будущая генерация видео
├─ social-publisher/     # будущая публикация в соцсети
└─ docs/                 # общая документация экосистемы
```

Проект сайта должен быть независимым от будущих сервисов. В дальнейшем он сможет быть связан с другими проектами экосистемы через API, ссылки, поддомены или общую авторизацию.

---

## 3. Цель текущего этапа

Разработать локальную версию сайта-блога, который позволяет:

- публиковать статьи;
- прикреплять изображения к статьям;
- распределять статьи по категориям и подкатегориям;
- назначать статьям теги;
- читать статьи последовательно;
- читать статьи выборочно по тегам, категориям и подкатегориям;
- использовать левое меню навигации;
- скрывать и показывать сайдбар;
- переключать светлую и тёмную тему;
- запускать сайт локально на компьютере разработчика.

### 4.2. Причина выбора Astro

Astro подходит для контентного сайта, потому что:

- хорошо работает со статьями;
- поддерживает Markdown и MDX;
- позволяет делать быстрые статические страницы;
- не требует сложной серверной инфраструктуры;
- подходит для локальной сборки;
- позволяет добавить интерактивные элементы только там, где они действительно нужны.

## 6. Контентная модель статьи

Каждая статья должна храниться в формате Markdown или MDX.

Пример файла статьи:

```md
---
title: "Введение в Запретные знания"
slug: "vvedenie-v-zapretnye-znaniya"
order: 1
category: "Основы"
subcategory: "Введение"
tags:
  - сознание
  - знание
  - путь
cover: "/images/articles/vvedenie.jpg"
description: "Вводная статья проекта «Запретные знания»."
published: true
date: "2026-06-29"
---

Текст статьи...
```

### Обязательные поля статьи

| Поле          | Назначение                                  |
| ------------- | ------------------------------------------- |
| `title`       | Название статьи                             |
| `slug`        | ЧПУ-адрес статьи                            |
| `order`       | Порядок статьи для последовательного чтения |
| `category`    | Основная категория                          |
| `subcategory` | Подкатегория                                |
| `tags`        | Список тегов                                |
| `cover`       | Изображение статьи                          |
| `description` | Краткое описание                            |
| `published`   | Публиковать или скрыть статью               |
| `date`        | Дата публикации                             |

## 7. Основные страницы сайта

### 7.1. Главная страница

Адрес:

```text
/
```

На главной странице должны отображаться:

- название проекта;
- краткое описание проекта;

- переход ко всем статьям;
- доступ к категориям;
- доступ к тегам.

---

### 7.2. Страница всех статей

Адрес:

```text

```

Функциональность:

- отображение названия, описания, категории, тегов и обложки;
- переход в конкретную статью.

---

### 7.3. Страница статьи

Адрес:

```text
/articles/[slug]
```

На странице статьи должны быть:

- заголовок статьи;
- изображение статьи;
- дата публикации;
- категория;
- подкатегория;
- теги;
- основной текст;
- навигация «предыдущая статья / следующая статья»;
- хлебные крошки;
- левый сайдбар с деревом статей;
- кнопка скрытия сайдбара;
- переключатель темы.

---

### 7.4. Страница категории

Адрес:

```text
/categories/[category]
```

Функциональность:

- вывод всех статей выбранной категории;
- отображение подкатегорий;
- переход в статьи;
- отображение тегов, относящихся к категории.

---

### 7.5. Страница подкатегории

Адрес:

```text
/subcategories/[subcategory]
```

Функциональность:

- вывод всех статей выбранной подкатегории;
- сортировка по `order`;
- переход в конкретную статью.

---

### 7.6. Страница тега

Адрес:

```text
/tags/[tag]
```

Функциональность:

- вывод всех статей с выбранным тегом;
- отображение количества статей;
- переход в конкретную статью.

## 8. Левый сайдбар

### 8.1. Назначение

Левый сайдбар является основным меню навигации по статьям.

### 8.2. Структура сайдбара

Сайдбар должен автоматически строиться на основе данных статей.

Пример:

```text
Основы
├─ Введение
│  ├─ Введение в Запретные знания
│  └─ Первые понятия
│
├─ Картина мира
│  ├─ Устройство реальности
│  └─ Человек и знание
│
Практики
├─ Наблюдение
├─ Внимание
└─ Внутренняя работа
```

### 8.3. Функции сайдбара

Сайдбар должен:

- показывать категории;
- показывать подкатегории;
- показывать статьи внутри подкатегорий;
- подсвечивать текущую статью;
- позволять переходить между статьями;
- иметь кнопку скрытия;
- иметь кнопку повторного открытия;
- корректно работать на мобильных устройствах.
- иметь кнопку развернуть и скрыть все категории, подкатегории, статьи

### 8.4. Состояние сайдбара

Сайт должен запоминать состояние сайдбара:

- открыт;
- скрыт.

Состояние можно хранить в `localStorage`.

## 9. Переключение светлой и тёмной темы

### 9.1. Требование

На сайте должен быть переключатель темы:

- светлая тема;
- тёмная тема.
- тема системы

### 9.2. Поведение

Пользователь должен иметь возможность переключить тему вручную.

Выбранная тема должна сохраняться после перезагрузки страницы.

Для хранения выбранной темы использовать `localStorage`.

### 9.3. Логика темы

При первом открытии сайта:

1. Если пользователь уже выбирал тему — использовать сохранённую тему.
2. Если сохранённой темы нет — можно использовать системную тему браузера.
3. Если системную тему определить нельзя — использовать светлую тему по умолчанию.

### 9.4. Техническая реализация

Рекомендуемый подход:

```text
html.light
html.dark
```

или:

```text
body.light-theme
body.dark-theme
```

Цвета должны быть вынесены в CSS-переменные.

Пример:

```css
:root {
  --color-bg: #ffffff;
  --color-text: #111111;
  --color-surface: #f4f4f4;
  --color-border: #dddddd;
}

html.dark {
  --color-bg: #101010;
  --color-text: #f5f5f5;
  --color-surface: #1a1a1a;
  --color-border: #333333;
}
```

### 9.5. Элементы, которые должны менять оформление

При смене темы должны корректно меняться:

- фон страницы;
- цвет текста;
- сайдбар;
- карточки статей;
- ссылки;
- кнопки;
- блоки кода;
- цитаты;
- хлебные крошки;
- навигация по статьям.

## 10. Навигация по статьям

### 10.1. Последовательное чтение

Статьи должны иметь поле `order`.

На странице каждой статьи должна быть навигация:

```text
← Предыдущая статья     Следующая статья →
```

Порядок определяется по значению `order`.

### 10.2. Выборочное чтение

Пользователь должен иметь возможность читать статьи выборочно:

- через категории;
- через подкатегории;
- через теги;
- через общий список статей;
- через сайдбар.

## 19. Итог

На выходе должен быть локально работающий сайт-блог «Запретные знания» на Astro, TypeScript, Markdown/MDX и Tailwind CSS.

Сайт должен быть подготовлен как самостоятельный проект внутри общей экосистемы, без привязки к Vercel и без реализации будущих AI/RAG-сервисов на текущем этапе.

Основной результат этапа — удобная локальная база для публикации и чтения статей с категориями, тегами, сайдбаром, последовательной навигацией и переключением светлой/тёмной темы.
